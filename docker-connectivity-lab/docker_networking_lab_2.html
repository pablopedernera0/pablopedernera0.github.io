🔴 Ejercicio 1: Volumen no Persistente

Un contenedor está usando datos de otro contenedor, pero cuando se reinicia, los datos desaparecen. Los volúmenes no están correctamente configurados.

🔧 Simular el problema

# Preparar el problema 1
sh problema-1-p2.sh

⚠️ Síntomas observados:

  • Los datos persisten después de crear un archivo
  • Al reiniciar el contenedor, se pierden los datos
  • No se usa volumen nombrado

🔍 Ejercicio 1: Diagnóstico

Paso 1: Crear archivo de prueba

docker exec data-app echo "datos-importantes" > /data/app.txt

Resultado esperado: Archivo creado

Paso 2: Verificar archivos en el contenedor

docker exec data-app ls -la /data/

Busca: ¿El archivo app.txt existe?

Paso 3: Listar volúmenes

docker volume ls

Busca: ¿Qué volúmenes están disponibles?

Paso 4: Inspeccionar el contenedor

docker inspect data-app | grep -A 15 Mounts

Busca: La sección Mounts - ¿hay volúmenes montados? ¿Qué tipo?

Paso 5: Detener y reiniciar


docker stop data-app
docker start data-app
docker exec data-app ls -la /data/

Busca: ¿El archivo sigue existiendo después de reiniciar?

🤔 Preguntas para reflexionar:

  • ¿El archivo desaparece después del reinicio?
  • ¿Qué tipo de almacenamiento está usando el contenedor?
  • ¿Cómo se puede hacer persistente el directorio /data?

🔴 Ejercicio 2: Puerto Publicado Incorrectamente

Un servicio está corriendo pero no es accesible desde el host, aunque el puerto parece estar mapeado.

🔧 Simular el problema

# Preparar el problema 2
sh problema-2-p2.sh

⚠️ Síntomas observados:

  • No se puede acceder al servicio desde localhost
  • El puerto parece estar mapeado
  • Dentro del contenedor funciona correctamente

🔍 Ejercicio 2: Diagnóstico

Paso 1: Verificar puerto del contenedor

docker exec web-service netstat -tlnp | grep LISTEN

Busca: En qué puerto está escuchando internamente

Paso 2: Ver mapping de puertos

docker port web-service

Resultado: Anota el mapeo de puertos

Paso 3: Verificar con netstat en el host

netstat -tlnp | grep LISTEN

Busca: ¿Está escuchando en 127.0.0.1 o en 0.0.0.0?

Paso 4: Intentar acceso desde host

curl http://localhost:9090

Resultado: ¿Funciona o da error de conexión?

Paso 5: Inspeccionar puerto en container

docker inspect web-service | grep -A 10 "PortBindings"

Busca: La dirección IP de binding - ¿es 127.0.0.1 o 0.0.0.0?

🤔 Preguntas para reflexionar:

  • ¿El puerto está mapeado a 127.0.0.1 en lugar de 0.0.0.0?
  • ¿Qué diferencia hay entre binding en localhost y en todas las interfaces?
  • ¿Cómo se remapea el puerto correctamente?

🔴 Ejercicio 3: Variables de Entorno No Heredadas

Una aplicación no está recibiendo las variables de entorno configuradas, causando fallos en la conexión a bases de datos.

🔧 Simular el problema

# Preparar el problema 3
sh problema-3-p2.sh

⚠️ Síntomas observados:

  • La aplicación falla sin conectarse a la BD
  • Las variables de entorno no son detectadas
  • Logs muestran variables undefined

🔍 Ejercicio 3: Diagnóstico

Paso 1: Listar variables de entorno

docker exec app-service env | grep -i database

Busca: ¿Aparecen las variables DATABASE_*?

Paso 2: Ver todas las variables

docker exec app-service env

Resultado: Lista completa de variables

Paso 3: Inspeccionar configuración del contenedor

docker inspect app-service | grep -A 20 "Env"

Busca: La sección Env en la salida

Paso 4: Ver los logs de la aplicación

docker logs app-service | head -20

Busca: Mensajes de error relacionados con conexión

Paso 5: Verificar archivo de configuración

docker exec app-service cat /app/config.sh

Busca: El nombre de la variable de entorno que se debe usar

🤔 Preguntas para reflexionar:

  • ¿Las variables de entorno están definidas en el contenedor?
  • ¿La aplicación las está buscando con el nombre correcto?
  • ¿Dónde se deberían definir estas variables?

🔴 Ejercicio 4: Límites de Recursos Excedidos

Un contenedor se detiene inesperadamente sin mensaje de error claro. Posiblemente alcanzó límites de recursos.

🔧 Simular el problema

# Preparar el problema 4
sh problema-4-p2.sh

⚠️ Síntomas observados:

  • El contenedor se detiene de forma inesperada
  • Sin mensajes de error en los logs
  • Status del contenedor es "Exited"

🔍 Ejercicio 4: Diagnóstico

Paso 1: Ver estado del contenedor

docker ps -a | grep worker-app

Busca: El status y el exit code

Paso 2: Ver logs detallados

docker logs worker-app

Busca: Mensajes de OOM o error killer

Paso 3: Inspeccionar limites de recursos

docker inspect worker-app | grep -A 5 "Memory"

Busca: Los valores de MemoryLimit

Paso 4: Ver estadísticas de uso

docker stats worker-app --no-stream

Busca: El consumo de MEM

Paso 5: Verificar OOMKilled

docker inspect worker-app | grep OOMKilled

Busca: El valor de OOMKilled - ¿es true o false?

🤔 Preguntas para reflexionar:

  • ¿Dice OOMKilled: true?
  • ¿Cuál es el límite de memoria asignado?
  • ¿Cómo se pueden aumentar los límites de recursos?

🔴 Ejercicio 5: Health Check No Configurado

Un contenedor parece estar corriendo pero la aplicación adentro no responde. No hay forma de saber si el servicio está realmente saludable.

🔧 Simular el problema

# Preparar el problema 5
sh problema-5-p2.sh

⚠️ Síntomas observados:

  • El contenedor aparenta estar "Up"
  • No hay información sobre salud del servicio
  • El servicio podría estar caído internamente

🔍 Ejercicio 5: Diagnóstico

Paso 1: Verificar status del contenedor

docker ps | grep health-service

Busca: El status completo y health

Paso 2: Inspeccionar health status

docker inspect health-service | grep -A 10 "Health"

Busca: La sección Health - ¿qué dice?

Paso 3: Intentar conexión al servicio

docker exec health-service curl -s http://localhost:3000/health

Busca: La respuesta del endpoint de health

Paso 4: Ver configuración del Dockerfile

docker inspect health-service | grep -i healthcheck

Busca: Si hay HEALTHCHECK configurado

Paso 5: Ver el intervalo de health check

docker inspect health-service | grep -A 5 "Health"

Busca: El campo "Status" - ¿es "none" o vacío?

🤔 Preguntas para reflexionar:

  • ¿Hay un health check configurado en el contenedor?
  • ¿Cuál debería ser el endpoint de health check?
  • ¿Cómo se configura un HEALTHCHECK en Docker?
